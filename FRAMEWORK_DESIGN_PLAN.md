# GoDoFramework 整体设计计划

> 文档定位：描述 GoDoFramework 为什么存在、要建设什么、按什么顺序建设，以及每一阶段如何验收。  
> 当前架构事实与已确定技术决策记录在 `ARCHITECTURE.md`；历史设想与痛点分析保留在 `GoDo_Framework_Overview.md`。

---

## 1. 框架愿景

GoDoFramework 是建立在 Godot 4.7 C# 之上的轻量游戏开发框架。它不替代 Godot，也不重新实现一套游戏引擎，而是把多个项目都会反复遇到的工程问题整理成稳定、可组合、可测试的通用能力。

框架的最终目的不是让所有代码都经过 GoDo，而是让开发者把主要注意力放在玩法、关卡、交互和内容上：

- Godot 负责节点、场景、资源、渲染、物理、音频和输入等引擎能力。
- GoDo 负责跨项目可复用的生命周期管理、通信、诊断和常用运行时服务。
- 具体游戏负责角色、战斗、数值、任务、剧情等业务逻辑。

判断框架是否成功，不看模块数量，而看它是否减少重复代码、隐藏的生命周期错误和跨系统耦合，同时仍然允许直接使用 Godot 原生 API。

---

## 2. 核心目标与非目标

### 2.1 核心目标

1. **减少重复工程工作**：把事件、错误处理、对象池、场景加载等通用机制沉淀为可复用模块。
2. **建立明确边界**：框架只提供机制，不包含任何具体游戏玩法。
3. **贴合 Godot 生命周期**：尊重场景树、Node、Resource、Signal 和主线程模型，不把 Godot 强行包装成另一套引擎。
4. **渐进采用**：项目可以只使用一个模块；可选模块不应成为其他模块的隐式前置条件。
5. **可诊断、可验证**：公开 API 有清晰的失败语义，关键行为有自动化测试或最小验证场景。
6. **性能可解释**：只对已知热点优化，避免无测量依据的复杂化；高频路径明确分配与生命周期成本。

### 2.2 非目标

- 不构建完整 ECS、依赖注入容器或新的场景系统来取代 Godot。
- 不在框架中实现角色控制、战斗、技能、任务、对话等玩法系统。
- 不保证每个模块都“零 GC”；正确性、清晰度和可维护性优先，高频路径再做针对性优化。
- 不一次性完成所有候选模块；只有被真实项目验证有重复价值的能力才进入稳定框架。
- 不承诺屏蔽所有 Godot 原生 API。框架必须始终允许安全逃脱到原生实现。

## 3. 设计原则

### 3.1 Godot 优先

优先组合 Godot 已有能力，再补齐工程缺口。Node 生命周期、Signal、Resource、Audio Bus 和线程化资源加载都应作为基础，而不是被无必要地重写。

### 3.2 机制与业务分离

`GoDo.*` 中不得出现只对某一种游戏成立的概念。框架可以定义事件通道和事件约束，但 `PlayerDiedEvent`、`EnemySpawnedEvent` 等具体事件属于游戏项目。

### 3.3 小核心、可选服务

核心只保留所有模块都需要的最小基础设施。Scene、Pool、Audio、Save、UI 等作为独立运行时服务存在，不因为目录命名而获得对其他模块的任意依赖权。

### 3.4 显式依赖

模块依赖必须能从构造、初始化或注册过程看出来。Service Registry 只作为业务层访问服务的边界，不允许成为框架内部隐藏依赖的捷径。

### 3.5 类型安全应落在正确位置

事件和服务接口适合使用泛型类型约束；场景和资源路径更适合由导出 `Resource`、资源目录或生成代码提供映射。不能为了表面上的泛型调用而隐藏真实资源路径和加载失败。

### 3.6 失败可见

参数错误、未注册服务、资源加载失败和生命周期误用必须有明确结果：抛出异常、返回 `Try...` 结果或形成结构化错误报告。每个 API 只选择一种清晰语义，避免静默失败。

### 3.7 先验证，再抽象

一个能力至少应在一个真实场景中出现明确痛点后再进入框架；在两个不同使用场景中得到验证后，才考虑标记为稳定公共 API。

## 4. 总体分层与依赖方向

```text
具体游戏业务
  角色 / 战斗 / 关卡 / 任务 / 游戏事件
                    │
                    ▼
GoDo Runtime Services（可选、彼此独立）
  Scene / Pool / Audio / Save / UI / Settings / Resources
                    │
                    ▼
GoDo Core（最小稳定核心）
  ErrorHandler / EventChannel / Service Registry / GoDoRuntime
                    │
                    ▼
Godot 4.7 C# / .NET
```

依赖规则：

1. 依赖只能整体向下，框架不得引用具体游戏程序集或业务类型。
2. Core 模块不通过 Service Registry 横向查找彼此。
3. Runtime Service 可以依赖 Core；服务之间如需直接协作，必须在设计阶段声明接口和依赖方向，不能默认全部改走全局事件。
4. EventChannel 用于一对多通知和解耦观察者，不用于替代所有方法调用，也不用于隐藏必须立即得到结果的请求。
5. ErrorHandler 是底层诊断出口，不反向依赖 EventChannel，避免错误处理形成循环。
6. Tools/Editor 工具只在编辑器中运行，不应成为运行时模块的依赖。

## 5. 模块规划

### 5.1 Core：最小稳定核心

#### EventChannel

用途：跨对象的一对多通知、生命周期绑定和纯 C# 订阅作用域。

保留方向：

- 泛型事件、`On` / `Off` / `Once` / `Emit`。
- Node 生命周期绑定与 `EventScope`。
- 派发期间安全增删、异常隔离和 Debug 诊断。

需要调整：

- Core 只保留 `IGameEvent` 和事件机制，不提供任何具体玩法事件。
- 游戏事件由具体游戏在自己的命名空间和目录中定义。
- 文档明确何时用直接调用、Godot Signal 或 EventChannel，避免事件总线泛滥。

#### ErrorHandler

用途：结构化错误报告、控制台输出和可插拔上报器。

当前代码已经具备分级、Reporter、`OnError`、Debug/Release 差异和 Autoload 异常入口，下一步应是稳定与验证，而不是重新设计。

后续重点：

- 验证递归上报、Reporter 异常、并发增删和退出阶段行为。
- `Fatal` 已明确为最高严重等级，不主动终止流程；是否退出由调用方决定。
- 远程 Reporter 只保留接口和示例，不在核心中绑定具体平台或第三方服务。
- 区分普通日志与错误报告；在真实需求出现前，不急于拆出庞大的 Log 子系统。

#### Service Registry

用途：向业务层暴露少量长期服务接口，并在测试中替换实现。

状态：暂缓开发。等 Scene、Audio 等首个需要业务层全局访问的服务出现后，再根据真实用法设计，避免先建立没有消费者的全局容器。

设计限制：

- 更准确的职责是“服务注册表”，不是通用依赖注入容器。
- 只注册明确接口，不鼓励任意类型成为全局单例。
- 必须提供重复注册、缺失注册、替换和清理的确定语义。
- 框架内部不使用它掩盖模块依赖。

#### GoDoRuntime

用途：管理框架启动、服务注册和退出清理的确定顺序。

当前由 Autoload 场景 `GoDoRuntime.tscn` 和脚本 `GoDoRuntime.cs` 提供，并保持职责很薄：

- 初始化 Core。
- 注册启用的可选服务。
- 退出时按逆序释放。
- 不承载游戏流程和具体玩法状态。

### 5.2 Runtime Services：按需启用

#### Pool（优先）

状态：首版基础实现已完成，位于 `GoDo/Runtime/Pool/`。

首版只支持 Godot Node/PackedScene 池：`NodePool<T>` 提供初始容量、空闲区容量、`Acquire/Release` 和 `IPoolable.OnAcquire/OnRelease` 回调。不要自动猜测需要重置的字段，重置责任由池化对象明确实现。

首版暂不支持纯 C# 对象池、多线程池、自动扫描和复杂淘汰策略。只有性能测量证明需要时再扩展。

#### Scene（优先）

首版聚焦：切换请求、线程化加载、进度、失败结果和可插拔过渡层。

需要先验证所谓 `Load<T>()` 是否真的能可靠映射脚本类型与 PackedScene。推荐以导出的场景目录 `Resource` 或显式 `SceneId -> PackedScene` 映射作为事实来源，避免把路径错误藏到泛型背后。

场景栈与 UI 栈是不同概念。首版只管理主场景切换；暂停菜单和弹窗优先留给 UI 模块。

#### Audio（中优先）

复用 Godot Audio Bus。首版提供 BGM 播放器、SFX 播放器池、总线音量接口和场景切换时的持续播放。

音频资源建议使用目录 Resource 或显式资源引用，不把任意字符串键散落在业务代码中。复杂混音、空间音频策略和语音系统后置。

#### Save（中优先）

框架只负责槽位、原子写入、备份、版本元数据、序列化接口和错误结果；具体存档数据结构与迁移规则属于游戏。

首版采用一种可靠格式即可，不同时开发 JSON、二进制和“加密”。加密不等于防作弊，不应作为基础承诺。

#### UI（低优先）

首版只解决层级根节点、界面打开/关闭、模态遮罩和返回栈。数据绑定和轻量 MVVM 暂不进入首版，因为它们容易扩张成另一套 UI 框架。

#### Settings（低优先）

游戏设置与进度存档分离。负责音量、显示、语言等用户偏好，并通过接口与 Audio 等模块协作；在 Audio 和 Save 基础稳定后再设计。

### 5.3 Tools：编辑器与开发期能力

#### Config

先支持 Godot `Resource` 作为强类型配置来源。只有实际项目确认需要表格工作流时，再增加 CSV/JSON 导入器；运行时热重载不是首版需求。

#### Extensions

只收录语义稳定、显著减少重复且不掩盖成本的扩展。Godot 已有同名或同等 API 时不重复包装，避免形成难以维护的杂物箱。

#### Diagnostics

包含事件监听统计、池占用、场景加载状态等 Debug 工具。诊断能力与对应模块一起建设，不先搭建独立“大而全”面板。

## 6. 暂缓或重新评估的模块

### Tick

不建议当前阶段把所有 `_Process` / `_PhysicsProcess` 统一接管。Godot 已经提供处理优先级、暂停模式和节点生命周期；集中 Tick 会增加注册成本、生命周期同步和调试间接性。

只有性能分析证明大量空转 `_Process` 是实际瓶颈，或项目确实需要分组更新频率时，再以可选调度器形式立项。Pool 和 Audio 不应为了未来的 Tick 提前产生依赖。

### Async 包装

不建设一套泛化协程框架。先为 Scene 等模块内部提供必要的 `ToSignal` 封装；只有重复模式稳定后才抽出公共 API。禁止为了异步而随意使用 `Task.Run` 操作 Godot 对象。

### StateMachine、Input、Localization、Timer

这些能力要么高度依赖具体游戏，要么 Godot 已有成熟基础。先通过真实项目验证框架层缺口，再决定是否加入；在此之前仅作为候选，不进入开发排期。

## 7. 分阶段开发路线

### 阶段 0：基线整理

目标：让文档、代码和边界一致。

- 明确 `FRAMEWORK_DESIGN_PLAN.md`、`ARCHITECTURE.md` 和 Overview 的职责。
- 更新 `ARCHITECTURE.md` 中模块真实状态。
- 将具体游戏事件移出 `GoDo` 框架命名空间；Core 只保留事件契约。
- 建立最小验证场景或测试目录，并记录执行方法。

验收标准：新开发者能够判断某段代码属于框架还是业务层；文档不再把已实现模块标记为待开发。

### 阶段 1：稳定 Core

目标：得到可长期依赖的小核心。

- 为 EventChannel 补齐关键派发、Once、生命周期和异常隔离测试。
- 为 ErrorHandler 补齐 Reporter、等级过滤和异常路径测试。
- Service Registry 延后到首个真实全局服务出现时设计。
- 继续保持 GoDoRuntime 初始化与释放职责的边界，不承载游戏流程。

验收标准：Core 可独立使用；无具体玩法类型；公共 API、失败语义和生命周期均有文档与验证。

### 阶段 2：解决高频运行时痛点

目标：以最小版本验证框架是否真正提升项目开发效率。

建议顺序：

1. Pool：首版基础实现已完成，下一步需要最小场景验证和压力测试。
2. Scene：先完成可靠加载和错误处理，再做过渡效果。
3. Audio：复用 Audio Bus，并以 SFX 池验证跨场景服务。

每个模块单独交付、单独验证，不等待整个阶段全部完成。

### 阶段 3：数据与产品能力

目标：处理项目长期运行产生的数据需求。

- Save：单一格式、版本信息、原子写入、备份恢复。
- Settings：与游戏进度分离的用户偏好。
- Config：优先 Resource 工作流，按需求增加导入工具。

### 阶段 4：界面与开发体验

目标：在真实项目需求明确后建设 UI 栈和诊断工具。

- UI 基础层级与模态栈。
- 模块 Debug 诊断视图。
- 经多处验证后再沉淀 Extensions。

### 阶段 5：候选能力评估

根据性能数据和至少一个真实项目的反馈，决定 Tick、状态机、输入封装、本地化辅助和更高级资源管理是否值得进入框架。

## 8. 单模块设计与验收模板

每个模块开始编码前，先完成以下最小设计：

1. **问题**：它解决了哪个重复痛点？原生 Godot 方案为什么不够？
2. **边界**：模块负责什么，明确不负责什么？
3. **调用者**：业务层、其他服务还是编辑器工具？
4. **依赖**：依赖哪些 Core 或 Godot 类型？是否形成循环？
5. **生命周期**：何时初始化、启用、禁用、释放？场景切换时怎样？
6. **失败语义**：异常、`Try` 返回还是错误报告？
7. **线程约束**：哪些操作必须在 Godot 主线程？
8. **性能预算**：高频路径在哪里？预期是否分配？如何测量？
9. **最小 API**：只列首版必需 API，不先设计所有未来扩展。
10. **验证方式**：自动化测试、最小场景和手动检查分别覆盖什么？

模块完成的统一标准：

- 不泄漏具体玩法概念。
- 不破坏既有 public API，或有明确迁移说明。
- 公开 API 有 XML 文档和使用示例。
- 生命周期订阅与释放对称，或使用已验证的自动绑定机制。
- Debug 信息足以定位失败，Release 不保留无意义高频诊断开销。
- 编译通过，并完成与风险匹配的自动化或编辑器验证。
- `ARCHITECTURE.md` 更新为实际状态，而不是继续保留过期计划。

## 9. 当前状态与近期建议

### 已有基础

- EventChannel 已完成边界清理与重入稳定化，可进入测试阶段。
- ErrorHandler、Reporter 接口和 GoDoRuntime 已存在；递归保护、监听者隔离与入口注册已完成，正式远程 Reporter 待后续实现。
- `NodePool<T>` 与 `IPoolable` 首版已实现，当前为实例级独立模块，不依赖 Service Registry。
- 已形成框架不感知具体玩法、Core 横向依赖受限的正确方向。

### 当前不一致

- Overview 提出“框架模块通过 ServiceLocator 互访”，而 Architecture 已改成仅供业务层使用；以后者为准。
- Overview 和 Architecture 都把 Tick 排得过早，目前没有证据证明需要集中接管所有更新。
- “泛型场景切换即可消灭路径”仍缺少可靠的类型到资源映射设计。

### 推荐的下一个具体任务

Pool 最小验证场景已覆盖初始化、复用、重复释放、空闲区容量和 `IPoolable` 生命周期；完成压力与异常路径测试后再进入 Scene 模块设计。

## 10. 文档维护规则

- `GoDo_Framework_Overview.md`：保留最初愿景、痛点和候选想法，作为历史输入，不要求持续同步实现状态。
- `FRAMEWORK_DESIGN_PLAN.md`：维护目标、模块范围、优先级和阶段路线；架构方向变化时更新。
- `ARCHITECTURE.md`：只记录当前已经采用的分层、依赖规则、公共模块和重要设计决策。
- `GODOT_GOTCHAS.md`：记录已实际遇到并确认的 Godot/C# 坑位。
- 每完成一个模块，先更新 `ARCHITECTURE.md` 的事实状态，再在本计划中更新阶段进度。

本计划不是不可变规范。真实项目反馈、性能数据和 Godot 版本变化都可以推动调整，但每次调整应说明问题、证据和取舍，避免仅凭“以后可能需要”扩大框架。
