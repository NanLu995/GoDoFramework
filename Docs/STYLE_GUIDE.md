# GoDoFramework 用户手册写作规范

用户手册面向第一次接触框架的人。目标是让读者由浅入深地完成真实任务，而不是把源码、接口清单或内部设计说明改写成网页。

## 内容边界

- 用户手册只写在 `Docs/Manual/<locale>/`。
- `AI/**` 与 `USAGE.md` 是事实来源和维护资料，永不直接发布，也不整段复制。
- public API 的精确签名、参数、返回值与异常放在 API Reference；手册只保留完成任务所需的调用。
- 内部架构只在会影响用户选择、配置或生命周期时解释。

## 先写读者任务

每页开头说明：读者会完成什么、适合什么场景、完成后能看到什么结果。正文优先采用以下顺序：

1. 前置条件。
2. 最小可运行步骤。
3. 可观察结果与验证方法。
4. 常见错误和修复方法。
5. 可选能力、进阶内容与 API Reference 链接。

不要从类型列表、目录结构或设计理念开始。首次出现框架术语时，用一句自然语言解释其作用。

## 页面类型

- 快速开始：用最短路径得到第一个可观察结果，不展开完整设计。
- 教程：按固定步骤从零完成一个完整目标，步骤有明确先后关系。
- 使用指南：帮助读者在不同方案间做选择，围绕一个用户任务组织。
- 概念说明：解释必须理解的机制、生命周期和边界，并链接到实际操作。
- 故障排查：按“现象 → 可能原因 → 确认方式 → 解决方法”组织。
- API Reference：由 XML 注释自动生成，供精确查询，不承担入门教学。

一页只承担一种主要职责。进阶指南和错误说明可以随着真实需求增加，不必为了覆盖源码目录机械创建页面。

## 示例与准确性

- 代码先由源码与编译确认，使用项目中真实存在的 API。
- 示例展示完整的最小调用边界；省略内容时明确标注。
- 围栏代码块必须写语言，例如 `csharp`、`powershell`、`json` 或 `text`。
- 操作会改变场景、项目配置或外部状态时，明确说明影响。
- 编译、自动测试和 Godot 手动验证分别描述，不用“已验证”笼统替代。
- 涉及输入、物理、节点生命周期、场景切换或编辑器配置时，写出 Godot 中的验证步骤。

## 标题、导航与链接

- 每个 Markdown 文件恰好包含一个 `# 一级标题`，标题层级不能跳级。
- 标题使用用户能理解的任务或概念，不依赖源码目录上下文。
- 新页面必须显式加入 `navigation.<locale>.json`，导航顺序代表推荐阅读顺序。
- 中英文页面使用相同相对路径；C# 标识符、资源键与命名空间保持源码拼写。
- 手册不复制完整 API 表格，使用链接引导读者查询 API Reference。

## 双语规则

中文是主要内容源，英文应保持事实、失败语义和验证范围一致，但不要求逐字直译。英文文件包含 `translation_of` 与 `translation_source_hash`；中文变化后必须先复核翻译，再更新摘要。

缺少翻译时不生成机器翻译占位内容。当前英文 API 页面复用中文 XML 注释，直到项目单独决定 API 注释语言策略。

## 功能变化时的维护顺序

用户可见功能新增或变化时：

1. 更新 public API XML 注释和对应 `USAGE.md` 技术事实。
2. 判断应更新教程、指南、概念说明、故障排查，还是仅需 API Reference。
3. 更新 `Docs/Manual/zh-cn/`；需要公开英文内容时同步复核翻译。
4. 更新 `coverage.json` 的状态、页面清单与契约摘要。
5. 运行 `python Docs/build_docs.py check`。

新增 `USAGE.md` 未登记、技术契约变化未复核、手册页面未加入导航或翻译过期，都会阻止检查通过。
